# 故障排查总览

## 这是什么
- 这组页面把排障分成两大类：设置 / 解析问题，以及数据同步 / 迁移问题。
- 这样做是为了避免把所有异常都塞进一个大杂烩列表里。

## 从哪里进入
- 当你遇到“没出卡”“没进队列”“改了设置没效果”“升级后状态怪怪的”等问题时。
- 当你已经完成基础阅读，想快速定位属于哪一层的问题时。

## 适合什么场景
- 你想先判断问题更像内容写法问题还是数据状态问题。
- 你需要一个按症状拆分的排障目录，而不是通读所有实现细节。
- 你准备把异常复现步骤整理给别人或留作自己以后参考。

## 具体步骤
1. 先问自己：这是“内容没被识别 / 设置没生效”还是“数据文件和界面看起来不一致”？
2. 如果偏向前者，进入 [设置与解析问题](./settings-and-parse-issues.md)。
3. 如果偏向后者，进入 [数据同步与迁移问题](./data-sync-and-migration-issues.md)。
4. 需要更底层的数据说明时，再回到 [数据与同步总览](../data-and-sync/index.md) 和 [附录](../appendix/index.md)。

## 相关设置 / 相关命令
- 排障前的基础知识： [设置总览](../settings/index.md)、[数据与同步总览](../data-and-sync/index.md)。
- 实验能力相关的额外边界见 [实验与高级](../experimental/index.md)。

## 常见错误
- 没先判断问题属于哪一层，就同时改设置、删文件、重建缓存。
- 只根据症状表面猜测原因，不保留复现步骤。
- 排障时直接在主库里做破坏性尝试，而没有备份。

## FAQ
- **为什么要分两类问题**：因为“文本没被识别”和“数据状态不同步”虽然症状可能相似，但根因和处理顺序完全不同。
- **排障一定需要看数据文件吗**：不一定。很多问题只靠内容样例和设置回顾就能解决。
- **什么时候该重建缓存**：只有当你已经判断它属于数据状态层面的问题时，才把它作为候选动作之一。

## 排错与风险提示
- 排障过程最怕没有边界；请先缩小问题层级，再决定是否要动数据文件或高级开关。
- 任何手工删改数据文件的动作都应该在完整备份之后进行。
