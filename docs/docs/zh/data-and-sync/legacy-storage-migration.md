# 旧存储迁移

## 这是什么
- 这一页只处理历史兼容和迁移，不把旧 `dataLocation` 或其他历史模式写成今天仍推荐的新方案。
- 它的重点是帮助你理解“为什么旧教程里说的路径或字段，现在看起来不一样了”。

## 从哪里进入
- 升级版本、导入旧数据、对照旧教程、查看兼容字段时。
- 当你发现代码或旧 JSON 里还出现历史字段时。

## 适合什么场景
- 你从旧版插件升级到当前实现，发现数据文件结构变了。
- 你读到老教程提到 `dataLocation`、旧 note 复习数据或旧算法字段。
- 你准备从旧库恢复状态，想知道哪些变化是迁移逻辑自动完成的。

## 具体步骤
1. 先接受一个主结论：当前正式支持的主口径是插件目录存储；旧模式只在迁移说明里保留。
2. 对照当前文件结构看哪些数据已经被拆分为独立文件，例如笔记复习和提交记录。
3. 如果你在旧数据中看到历史字段，不要急着手改；先确认当前版本是否会在加载时自动迁移或收敛这些字段。
4. 迁移前一定做完整备份；迁移后再对照当前文档确认哪些设置和值已经不该继续手工维持。

## 常见错误
- 把旧教程里的路径和字段当成当前推荐方案照抄。
- 在不了解自动迁移逻辑的前提下手工改动旧数据。
- 升级后只检查一个文件，忽略了数据已经被拆分到多个文件。

## FAQ
- **旧 `dataLocation` 现在还能当主方案吗**：文档不再把它作为正式推荐方案；它属于历史兼容说明。
- **为什么升级后文件变多了**：因为不同类型的数据已经被拆分管理，以便更清晰地处理卡片、笔记复习、提交记录和 AI 主题包。
- **迁移是不是一定会破坏旧数据**：正常情况下迁移的目标是保留它，但你仍然应该在升级前做完整备份。

## 排错与风险提示
- 最大的风险不是迁移本身，而是不做备份就手工尝试各种旧字段和旧路径的组合。
- 如果你需要长期保留历史库，最好把“可运行旧版本”和“当前版本迁移后状态”分开保存，避免互相覆盖。
