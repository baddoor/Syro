## 这是什么
- 这一页专门解释闪卡的 FSRS 与笔记复习的 WMS（Weighted Multiplier）相关参数。
- 文档会明确区分“当前真实生效的调度参数”和“更像模拟器或预留项的 UI 参数”，避免把未完全接线的控件写成正式承诺。

## 从哪里进入
- Algorithm 标签页里的 FSRS 区域。
- Algorithm 标签页里的 WMS 模拟器与参数区域。

## 适合什么场景
- 你已经稳定使用插件一段时间，想进一步调节节奏。
- 你发现某些参数名字很像会影响调度，但改完以后体感并不明显。
- 你需要把算法解释给其他用户，避免把模拟器和真实调度混在一起。

## 具体步骤
1. 先把 FSRS 和 WMS 看成两条不同主线：闪卡主要看 FSRS，笔记复习主要看 WMS。
2. 在 FSRS 区域优先关注目标记忆保持率、最长间隔、是否启用 fuzz、是否启用短期调度等真正会改变调度行为的设置。
3. 在 WMS 区域先认清当前真实生效的核心是重要性映射，也就是 `impMin` 和 `impMax` 这类参数。
4. 如果你看到 Again / Hard / Good / Easy 一组 WMS 相关控件，先把它们理解成更偏模拟器 / 预留项的参数，不要直接假设它们已经完整改写真实调度公式。

## 常见错误
- 把 FSRS 和 WMS 混成一套统一公式来理解。
- 只看 UI 名称，不看当前实现是否真的把那个控件接进了真实调度路径。
- 同时大幅调整多组参数，最后不知道是哪一项真正导致节奏变化。

## FAQ
- **为什么这里会反复强调“真实生效参数”**：因为当前实现里确实存在一部分更像模拟器或预留项的控件；文档必须把它们和稳定参数分开。
- **新用户应该先调算法吗**：通常不应该。先把内容写法、同步和队列习惯跑顺，再决定是否需要调算法。
- **如果我改了参数却几乎没感觉，是不是坏了**：不一定。可能你改的是体感不明显的参数，也可能你碰到的是更像模拟器 / 预留项的控件。

## 排错与风险提示
- 算法页最容易让人误以为“所有控件都同样重要”。实际上你应该优先关注少数真正生效的核心参数。
- 如果你准备根据算法行为做迁移或比较版本差异，先备份数据，并同时查看 [数据与同步](../data-and-sync/index.md)。
