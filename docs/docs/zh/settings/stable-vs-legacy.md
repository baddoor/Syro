# 稳定设置与兼容字段

## 这是什么
- 这一页先帮你区分三类内容：当前 UI 可见且建议普通用户理解的稳定设置、代码中仍保留但不建议依赖的兼容字段、以及更像预留项或内部状态的参数。
- 只要这一步分清，后面的算法页和数据页就会容易很多。

## 从哪里进入
- 当前设置页的 Flashcards、Notes、Algorithm、UI、Sync 标签。
- 历史配置迁移时保留下来的旧字段和别名字段。

## 适合什么场景
- 你在旧教程、旧 issue 或历史截图里看到一些当前 UI 没有的字段。
- 你想知道某个字段为什么在代码里还存在，却不该作为日常主配置来写。 
- 你需要给其他用户解释“当前推荐怎么配”和“历史兼容为什么还保留”。

## 具体步骤
1. 把“当前 UI 可见、并且对日常工作流有直接影响”的设置视为稳定设置，例如卡片顺序、Cloze 相关开关、队列过滤、状态栏和同步提示等。
2. 把“代码里仍然保留、但当前 UI 不一定完整暴露”的字段视为兼容字段，只在迁移或排障时关注。
3. 当一个控件名称看起来很重要，但文档反复提示“更像模拟器 / 预留项”时，不要把它当作正式承诺的配置面板。
4. 如果你不确定某项应该归哪类，优先查 [算法与 WMS](./algorithms-and-wms.md) 或 [数据与同步](../data-and-sync/index.md)。

## 相关设置 / 相关命令

## 常见错误
- 把旧 `dataLocation` 等历史字段写成今天仍推荐的新手选项。
- 看到兼容字段就手工改 JSON，却不知道迁移逻辑会在升级时再次收敛它。
- 只因为 UI 里看到了控件，就假设它一定已经完整驱动真实调度。

## FAQ
- **为什么不把所有字段都写进主教程**：因为那会让普通用户分不清哪些是稳定功能，哪些只是兼容历史。
- **兼容字段是不是完全不能碰**：不是。它们通常用于迁移、排障或理解旧数据，而不是作为主工作流的推荐配置。
- **我怎么判断一个设置值是否真的在当前实现里生效**：看文档是否明确把它标记为稳定生效项；如果没有，就按更保守的口径理解。

## 排错与风险提示
- 兼容字段最容易让人误以为“还有另一套正式工作流”；如果你在写教程或给别人配置，请优先引用稳定设置页。
- 手工编辑旧字段前，一定先备份数据文件。迁移逻辑和当前 UI 可能不会按你想象的方式持续尊重这些值。
